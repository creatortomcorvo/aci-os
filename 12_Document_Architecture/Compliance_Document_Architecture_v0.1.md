# Compliance Document Architecture v0.1

Status: active core architecture artifact.

Source: founder note on durable compliance-document architecture, reviewed 2026-07-04.

Purpose: define how ACI-OS should understand, assess, draft, and improve Codes of Conduct, policies, procedures, work instructions, guidelines, forms, registers, dashboards, local addenda, and evidence records.

This is not a one-off drafting note. It is a document operating-system model for a compliance program.

## Core Doctrine

The Code of Conduct is the foundation of the compliance program. Policies translate the Code into mandatory standards. Procedures translate policies into controlled decision-making. Work instructions translate procedures into daily tasks. Forms, registers, and systems create evidence. Monitoring, audits, investigations, and management review ensure the system remains effective, current, and proportionate to risk. Advice, approval, legal decision, and management risk decision are separate roles and must not be confused.

Short version:

Values -> obligations -> risks -> rules -> decisions -> controls -> evidence -> monitoring -> improvement.

Even shorter:

- Code says what we stand for.
- Policies say what must or must not happen.
- Procedures say how decisions are made.
- Work instructions say exactly how tasks are performed.
- Forms and systems create evidence.
- Monitoring checks whether reality matches the documents.
- Investigations and audits improve the system.

## Design Principle

A strong compliance documentation system is not a pile of documents. It is a controlled operating system.

The most important structural rule:

Do not let one document do every job.

A Code inspires and anchors. A policy sets rules. A procedure controls decisions. An instruction guides tasks. A form captures facts. A register proves activity. A dashboard shows performance. An audit tests reality. An investigation teaches the system where it failed.

## Source Anchors

Use these as source anchors when building source-backed versions, subject to current-version verification before external use:

- ISO 37301 for compliance management system architecture: obligations, risk assessment, leadership, compliance policy, roles, documented information, operational controls, raising concerns, investigations, monitoring, internal audit, management review, and continual improvement.
- Anti-bribery management-system standards, including current ISO anti-bribery guidance where verified, for anti-bribery policy and control structure.
- DOJ ECCP / FCPA enforcement expectations for well-designed, resourced, empowered, and effective-in-practice compliance programs.
- UK Bribery Act guidance for proportional procedures, top-level commitment, risk assessment, due diligence, communication/training, and monitoring/review.
- U.S. Sentencing Guidelines logic for compliance-and-ethics governance, prevention, detection, reporting, discipline, monitoring, and periodic effectiveness evaluation.
- Founder AD1-AD4 methodology for document classification and avoiding policy/procedure/template confusion.

## Document Hierarchy

| Level | Document type | Main question answered | Primary audience |
| ---: | --- | --- | --- |
| 0 | Document Governance Standard | How are compliance documents created, approved, reviewed, owned, versioned, and retired? | Compliance, Legal, document owners |
| 1 | Code of Conduct | What kind of company are we, and what behavior is expected from everyone? | All employees, leaders, relevant third parties |
| 2 | Compliance Management Policy / Program Charter | How is the compliance program governed and operated? | Board, management, Compliance, Legal, Audit |
| 3 | Topic Policies | What is required or prohibited in a risk area? | Employees, business owners, relevant third parties |
| 4 | Procedures | How exactly do we apply the policy and make decisions? | Process users, approvers, control owners |
| 5 | Processes / Workflows | How does work move across functions and systems? | Business, Finance, HR, Procurement, IT, Compliance |
| 6 | Work Instructions / SOPs | What exact task must a user perform? | Operational users |
| 7 | Guidelines / FAQs / Playbooks | How should rules be interpreted in difficult cases? | Business, managers, Compliance advisers |
| 8 | Forms / Templates / System Fields | What information and evidence must be captured? | Requesters, approvers, investigators |
| 9 | Registers / Logs / Dashboards | What records prove the system is working? | Compliance, Audit, management |
| 10 | Local Addenda | What stricter or local legal requirement modifies the global rule? | Local Legal, local Compliance, local management |

## Dependency Chain

Every important compliance rule should have lineage.

Example:

Code of Conduct -> Anti-Bribery and Corruption Policy -> Gifts, Hospitality and Travel Procedure -> Hospitality Approval Form -> Gifts and Hospitality Register -> Quarterly Monitoring Report -> Corrective Action Tracker.

A weak program says: "We have a policy."

A strong program says: "We have a Code commitment, a policy rule, a procedure for decisions, a form that captures facts, a register that stores evidence, monitoring that tests it, and a corrective-action process when it fails."

## One Obligation Rule

Every compliance obligation must have:

- One owner.
- One rule.
- One control.
- One evidence source.
- One monitoring method.
- One escalation path.

If a rule has no owner, it is a slogan.

If a rule has no control, it is a wish.

If a rule has no evidence, it is unverifiable.

If a rule has no monitoring, it will decay.

## Build Sequence

For a durable template pack, build in this order:

1. Document Governance Standard.
2. Compliance risk universe and obligation register.
3. Code of Conduct.
4. Compliance Management Policy / Program Charter.
5. Core risk policies.
6. Procedures.
7. Work instructions.
8. Guidelines and FAQs.
9. Forms, registers, and system fields.
10. Training and communications.
11. Monitoring, audit, investigation, and improvement loop.

The Code of Conduct is the first document people should see. It is not necessarily the first document the project team should draft. The team should first define the hierarchy, governance rules, risk universe, mandatory topic-policy list, decision-rights model, escalation architecture, and local addendum model.

## Document Type Rules

### Code of Conduct

The Code is the visible constitution of the compliance program. It should be clear, concise, accessible, translated where needed, periodically reviewed, and connected to deeper policies and procedures.

Must include:

- CEO / Board message.
- Scope: employees, directors, officers, contractors, agents, subsidiaries, and relevant third parties.
- Values and decision principles.
- Ethical decision test.
- Advice does not equal approval; approval does not equal management risk decision.
- Main risk chapters.
- Speak Up and non-retaliation.
- Consequences.
- Where to find more.
- Document owner and review cycle.

### Policy

A policy defines the mandatory rule system. It should not describe every workflow.

Must include:

- Document control.
- Purpose.
- Scope and applicability.
- Policy statement.
- Definitions needed for decisions.
- Core rules.
- Roles and responsibilities.
- Approval and escalation model.
- Records and evidence.
- Exceptions.
- Speak Up / reporting where relevant.
- Consequences.
- Training and communication.
- Monitoring and review.
- References to Code, procedures, guidelines, laws, or standards.

A policy must not become a legal memo, hidden procedure, training deck, vague aspiration, rule without owner, rule without evidence, prohibition without escalation path, or global rule that ignores mandatory local law.

### Procedure

A procedure is where the program becomes operational.

It answers: what happens first, next, who decides, by when, using what evidence, and what if the answer is no?

Must include:

- Purpose.
- Trigger.
- Inputs.
- Scope.
- Roles.
- Process steps.
- Decision points with explicit outcomes.
- Risk rating.
- Escalation triggers.
- Controls.
- Evidence.
- Timelines / SLAs.
- Exceptions.
- Outputs.
- Monitoring.
- Appendices: flowchart, RACI, templates, examples, FAQ.

Every procedure should label outputs:

- Advice only.
- Recommendation.
- Compliance approval.
- Legal decision.
- Management risk decision.
- Escalation mandatory.
- Stop / prohibited.

This prevents "Compliance advised" from becoming "Compliance approved."

### Work Instruction / SOP

A work instruction is task-level.

It answers: exactly what do I click, enter, upload, check, save, or send?

Must include:

- Task name.
- User role.
- When to use this instruction.
- Before you start.
- Step-by-step actions.
- Screenshots or system-field guidance where available.
- Mandatory fields.
- Common errors.
- What good evidence looks like.
- Where to save records.
- Escalation contact.
- Version and system owner.

### Guideline / FAQ / Playbook

Guidelines are for judgment, examples, interpretation, and learning. They should not create new mandatory rules unless clearly approved as binding.

Must include:

- Purpose.
- Link to controlling policy.
- Plain-language explanation.
- Good / bad examples.
- Decision trees.
- Red flags.
- Common pressure phrases.
- Suggested business responses.
- When to escalate.
- FAQ.
- Country or business-line examples.
- Related forms and contacts.

### Forms, Registers, Dashboards

Forms and systems are controls because they create evidence.

They should define:

- Required fields.
- Evidence attachments.
- Ownership.
- Retention.
- Access.
- Review status.
- Approval conditions.
- Exception markers.
- Monitoring fields.
- Link to the relevant policy/procedure.

### Local Addendum

Global companies should usually use:

Global Policy + Local Addendum + Local Procedure where needed.

Local addenda may raise standards or adapt to mandatory law. They should not weaken the global standard without Legal and Compliance approval.

Must include:

- Country / entity.
- Global policy reference.
- Local legal owner.
- Local compliance owner.
- What is stricter locally.
- What is prohibited locally.
- Local thresholds.
- Works council / employee notice requirements where relevant.
- Local reporting channels.
- Local record retention.
- Local approval deviations.
- Local regulator / authority references.
- Effective date.
- Review cycle.

## Human-Usability Rule

A good compliance document should be written for the person under pressure.

For each policy or procedure, include a short box:

"When under pressure, remember..."

Examples:

- Do not approve verbally.
- Do not split value to avoid thresholds.
- Do not rely on "everyone does it."
- Do not proceed if a tender, renewal, permit, inspection, audit, or dispute is pending.
- Do not onboard a third party because the customer recommended them.
- Do not pay offshore without documented rationale.
- Do not accept vague services.
- Do not approve success fees without proportionality and deliverables.

This is where the human layer of compliance belongs.

## Lifecycle

Every document should move through a controlled lifecycle:

1. Need identified.
2. Owner assigned.
3. Baseline selected.
4. Draft prepared.
5. Legal / Compliance review.
6. Business consultation.
7. Control / evidence review.
8. Approval.
9. Publication.
10. Training / communication.
11. Attestation if needed.
12. Monitoring.
13. Periodic review.
14. Revision.
15. Retirement / supersession.

Review triggers:

- Law or regulator expectation changes.
- Source standard or guidance changes.
- Risk assessment changes.
- Investigation reveals a root cause.
- Audit finds a gap.
- Repeated exceptions occur.
- Business model changes.
- New technology is introduced.
- Acquisition or restructuring occurs.
- Country expansion occurs.
- Employees repeatedly misunderstand the rule.

## Review Checklist

Before approving a Code, policy, procedure, instruction, or guideline, ask:

- What risk does this document control?
- Which Code principle does it implement?
- Which obligation does it map to?
- Who owns it?
- Who approves it?
- Who must follow it?
- What decision does it help make?
- What evidence does it create?
- What system or register supports it?
- What happens if the rule is breached?
- What happens if facts are unclear?
- What cannot be locally approved?
- What requires Legal or external counsel?
- What training is required?
- How will we test whether it works?
- When must it be reviewed?
- What document does it replace?
- Is it readable by the intended user?

## Common Design Mistakes

- Policy written like a procedure.
- Procedure with no decision rule.
- Code that is too legalistic.
- No local addendum model.
- No evidence design.
- No owner.
- No review triggers.
- No connection to training, communication, monitoring, and improvement.

## ACI-OS Product Implications

When ACI-OS handles document requests, it should not merely draft text. It should first classify the document type and ask:

- Is this a Code, policy, procedure, process, work instruction, guideline, form, register, dashboard, or local addendum?
- What parent document controls it?
- What child documents, forms, systems, or evidence records must connect to it?
- What obligation, risk, control, evidence source, monitoring method, and escalation path does it create?
- Is the user asking for advice, recommendation, Compliance approval wording, legal decision support, management decision support, or escalation?

The product should refuse to let one document do every job. If a user asks for a "policy" but describes step-by-step workflow, ACI-OS should say: "That belongs in a procedure." If a user asks for a "procedure" but provides only values, ACI-OS should say: "That belongs in the Code or policy."

## Act / Correct / Park / Discard Verdict

Act:

- Add document architecture as a core repository area.
- Adopt the document hierarchy, dependency chain, one-obligation rule, lifecycle, review checklist, local-addendum model, and human-usability box.
- Connect advice / approval / legal decision / management risk decision labels to procedures and approval workflows.

Correct:

- Do not treat "approval note" as ACI-OS approval. It may be documentation for a verified authorized owner.
- Do not use source-version claims externally without checking current versions.
- Keep document-management software and workflow approvals out of Sprint 1 implementation, while preserving the architecture for future product behavior.

Park:

- Full source-backed templates for every document type.
- Detailed sports/media/sponsorship template pack.
- Document-management module, workflow approvals, audit-log implementation, and enterprise repository integrations.

Discard:

- Any approach that treats document drafting as isolated text production.
- Any architecture where a Code, policy, procedure, instruction, form, register, or dashboard is asked to do another document type's job.
